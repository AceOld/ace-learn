1. Parquet 凭什么碾压 CSV(10 分钟)

CSV/JSON 是行式 + 文本。Parquet 是列式 + 二进制 + 带统计。这四个特性叠加,使分析查询快 10–100 倍、省 5–20 倍存储。

机制原理对查询的影响
列式存储 同列同类型数据连续存放 压缩率高(类型一致)+ 列裁剪(SELECT a,b 只读 a,b 两列)
Row Group 统计 每 ~128MB row group 存 min/max/count/bloom 谓词下推:WHERE 跳过整个 row group 不读
向量化布局 数据按 1024/2048 行的 page 组织 批量读取 → SIMD 指令加速(引擎吃满 CPU)
不可变(immutable) 写一次读多次,不修改 适合 WORM(写一次读多次),不适合频繁 upsert
⚠ 关键认知

同一份 parquet,不同引擎能榨出的性能差异可达 10 倍。差别在引擎有没有好好用上面四个特性。下面讲的就是怎么选对引擎。

Q1 · 同样一份 10GB parquet,两个引擎都能跑 SELECT a,b WHERE c>100,但引擎 A 比 B 快 5 倍。最可能的原因?
  • A 的 CPU 更快
  • A 用了更快的磁盘
  • A 做了谓词下推和列裁剪
  • A 偷懒只读了 2GB
答案:C。同一份文件、同一硬件,差异来自引擎的查询优化:列裁剪(只读 a/b/c 不读其他几十列)+ 谓词下推(读 row group 统计,跳过 c≤100 的 row group)。这就是为什么"parquet 是基础,引擎是关键"。选错引擎,parquet 的优势吃不到一半。A/B 是硬件层面(题设说同一硬件);D 是误读。

⚠ ClickHouse 的关键澄清

ClickHouse 主力存储是自研 MergeTree(不是 parquet)。它读 parquet 时用 row group 统计做谓词下推,但真正的数据跳过神技(skip indexes:minmax/bloom)只在 MergeTree 表上生效。如果你拿一堆 parquet 给 ClickHouse 查,它的优势打折 —— 不如直接导进 MergeTree。

定位边界(常被混淆的三者)

工具思维范式本质适合谁
DuckDBSQL 优先完整数据库(ACID/MVCC/扩展)想用 SQL 跑分析的
Polars代码优先(DataFrame)库,无持久化/事务从 Pandas/Spark 来的
chDBSQL + ClickHouse 方言嵌入式 CH(同内核)已在 CH 生态、想本地化

一句话区分:SQL 思维 → DuckDB;代码/DataFrame 思维 → Polars;已在 ClickHouse 生态 → chDB。这三个互相覆盖 70%,但各自有不可替代的 30%。

3. 引擎对 parquet 特性的利用度(15 分钟)

不是所有引擎都完整利用 parquet 的特性。下面是核实过的事实(2026-07)。

特性DuckDBClickHousePolarsDataFusion
列裁剪✅✅✅✅
谓词下推(row group 统计)✅✅✅(scan_parquet)✅
Page 级谓词下推✅部分部分✅
Bloom filter✅ 读+写(主要 MergeTree)❌✅
向量化执行✅(2048)✅✅✅
Metadata 缓存✅✅部分✅(50.0+)
⚠ Polars 最常见的反模式

pl.read_parquet('f.parquet').lazy().filter(...) 是错的 —— 这会先全量读再过滤,谓词下推失效。正确写法: 

# ❌ 错:全量读再过滤
df = pl.read_parquet('f.parquet').lazy().filter(pl.col('c') > 100)

# ✅ 对:扫描时下推过滤
df = pl.scan_parquet('f.parquet').filter(pl.col('c') > 100)

什么时候 parquet 反而是错的选择?

场景为什么 parquet 不合适替代
小数据(<100MB)row group 元数据开销超过数据本身CSV / JSON / SQLite
频繁 upsertparquet 不可变,改一行重写整文件Iceberg / Delta Lake
OLTP(事务)列存不适合点查点写PostgreSQL / MySQL
流式写入parquet 需缓冲成 row group 才能写Avro + Kafka
整行读取(全列)列存没优势,反而有开销Avro / JSON
Q2 · ClickHouse 的"数据跳过神技"(skip indexes)在什么场景下真正生效?
  • 查任意 parquet 文件
  • 查 ClickHouse 自己的 MergeTree 表
  • 查 CSV 文件
  • 查 JSON 文件
答案:B。ClickHouse 的 minmax/set/bloom_filter 等 skip indexes 只在 MergeTree 表族上生效,不在任意 parquet 文件上。这是常被混淆的点 —— 很多人以为"ClickHouse 查 parquet 也快",其实它读 parquet 时只做了 row group 统计下推,真正的神技在 MergeTree 表上。所以用 ClickHouse 的最佳姿势是把 parquet 导入 MergeTree,而不是直接查 parquet。

4. 可运行 Benchmark:DuckDB vs Polars vs chDB(20 分钟)

准备数据(2 分钟)

pip install duckdb polars chdb pyarrow

# 用 DuckDB 生成 1000 万行测试 parquet
python -c "
import duckdb
con = duckdb.connect()
con.execute('''
  COPY (
    SELECT
      range AS id,
      random() * 1000 AS amount,
      'user_' || cast(range % 10000 AS varchar) AS user_id,
      cast('2026-01-01' AS date) + INTERVAL (range % 365) DAY AS dt
    FROM range(10000000)
  ) TO 'test.parquet' (FORMAT parquet, ROW_GROUP_SIZE 100000);
''')
print('生成完成')"

Benchmark A:DuckDB(2 分钟)

import duckdb, time
t0 = time.time()
df = duckdb.sql("""
  SELECT user_id, sum(amount) AS total, count(*) AS cnt
  FROM 'test.parquet'
  WHERE dt >= '2026-06-01' AND amount > 500
  GROUP BY user_id ORDER BY total DESC LIMIT 10
""").df()
print(f"DuckDB: {time.time()-t0:.2f}s")
print(df.head())

Benchmark B:Polars(正确写法)(2 分钟)

import polars as pl, time
t0 = time.time()
df = (
    pl.scan_parquet('test.parquet')                      # ✅ scan 不是 read
      .filter((pl.col('dt') >= pl.date(2026,6,1)) &
              (pl.col('amount') > 500))                   # 谓词下推
      .group_by('user_id')
      .agg(pl.sum('amount').alias('total'),
           pl.count().alias('cnt'))
      .sort('total', descending=True)
      .head(10)
      .collect()                                          # 触发执行
)
print(f"Polars (正确): {time.time()-t0:.2f}s")

Benchmark C:chDB(嵌入式 ClickHouse)(2 分钟)

import chdb, time
t0 = time.time()
res = chdb.query("""
  SELECT user_id, sum(amount) AS total, count(*) AS cnt
  FROM file('test.parquet', Parquet)
  WHERE dt >= '2026-06-01' AND amount > 500
  GROUP BY user_id ORDER BY total DESC LIMIT 10
""")
print(f"chDB: {time.time()-t0:.2f}s")
print(res)
预期结果(1000 万行 × 4 列,普通笔记本)

DuckDB / Polars(正确)/ chDB 三者通常同一数量级(亚秒到秒级),具体取决于 CPU 核数和磁盘速度。这个 benchmark 不是为了找"最快",是为了让你看清三者写法差异 + 谓词下推的效果。

练习 10.1 · 跑反模式对比 10 min

把 Polars 的代码改成反模式:

# ❌ 反模式:先 read 再 filter(谓词下推失效)
df_bad = pl.read_parquet('test.parquet').lazy() \
    .filter((pl.col('dt') >= pl.date(2026,6,1)) &
            (pl.col('amount') > 100)) \
    .group_by('user_id').agg(pl.sum('amount').alias('total')) \
    .collect()

对比 read_parquet().lazy() 和 scan_parquet() 的耗时差。这个差距就是谓词下推的价值,也解释了为什么"用对工具"比"用快工具"更重要。

5. 写 parquet 的调优(5 分钟)

参数影响建议
ROW_GROUP_SIZE太小:每 group 元数据开销大,5–10× 慢;太大:内存爆100K–1M 行/group
COMPRESSIONsnappy:快、文件大;zstd:慢一点、文件小 25–40%热数据 snappy / 冷数据 zstd
列顺序过滤频繁的列放前面,row group 统计更有效按"过滤频率"排
分区按 dt 分目录,谓词下推直接跳目录常按 dt 分区
 
-- DuckDB 写 parquet 的推荐参数
COPY (SELECT * FROM source) TO 'out.parquet' (
  FORMAT parquet,
  COMPRESSION zstd,             -- 冷数据用 zstd
  ROW_GROUP_SIZE 100000         -- 10 万行/group
);

-- 分区写入(按 dt)
COPY (SELECT * FROM source) TO 'partitioned/' (
  FORMAT parquet,
  PARTITION_BY (dt)             -- 生成 partitioned/dt=2026-07-01/...
);

6. 选型决策树(10 分钟)

把上面的所有维度压成一张决策表。回答下面五个问题,就能定。

问题答案 → 选
1. SQL 还是代码思维?SQL → DuckDB / chDB;代码 → Polars
2. 数据量?GB → DuckDB/Polars;TB+ 单机 → ClickHouse server
3. 并发?单用户 → 嵌入式;多用户 → ClickHouse server
4. 需要 upsert/事务?要 → Iceberg/Delta on parquet;不要 → 纯 parquet
5. 已在哪个生态?CH 生态 → chDB/server;Arrow 生态 → DataFusion
三条决策捷径
  1. 90% 的单机分析场景:DuckDB 一把梭(SQL 友好、parquet 一等公民、零依赖)。
  2. 已在 ClickHouse 生态:本地 chDB,生产 CH server。
  3. 从 Pandas/Spark 来的、代码思维:Polars(惰性 + 谓词下推)。
Q3 · 单人分析师,10–50GB parquet 数据,主要写 SQL 查询,偶尔 Pandas 接 ML。最佳选择?
  • ClickHouse server(强但过重)
  • Spark(分布式但单机杀鸡用牛刀)
  • DuckDB(嵌入式 SQL,parquet 一等公民)
  • 纯 Pandas(全量读内存,parquet 优势吃不到)
答案:C。单用户 + SQL 思维 + GB 级 parquet = DuckDB 教科书场景:嵌入式零依赖、SQL 直接查 parquet、列裁剪+谓词下推全开、内存不够能 spill。A 太重(单人为查询起 server);B 单机用 Spark 是反模式;D 全量读内存既慢又撑不下 50GB。DuckDB 还能通过 df = duckdb.sql('...').df() 直接和 Pandas 无缝衔接。
Q4 · 你的场景需要频繁 upsert(订单状态会变),应该用?
  • 纯 parquet,定期重写整文件
  • Iceberg 或 Delta Lake(在 parquet 上加 ACID)
  • CSV(行式好改)
  • JSON(灵活)
答案:B。parquet 不可变,upsert 单行要重写整文件 = 灾难。Iceberg / Delta Lake / Hudi 是"表格式",在 parquet 文件之上加 ACID + 时间旅行 + 高效 upsert,正是为这个场景设计的。A 是反模式;C/D 是退回原始格式(分析性能崩)。这就是为什么 L09 的 DWD 层在需要 SCD2 时,常在 Iceberg/Delta 之上实现。

本课要点回顾

  1. Parquet 四机制:列存 + row group 统计 + 向量化 + 不可变。
  2. 同一份 parquet,引擎选错差 10×:谓词下推/列裁剪是关键。
  3. 2026 单机 OLAP 五引擎:DuckDB(SQL 嵌入式)/ ClickHouse(server)/ Polars(DataFrame)/ chDB(嵌入 CH)/ DataFusion(Rust 引擎)。
  4. ClickHouse 神技只在 MergeTree:不是任意 parquet。
  5. Polars 必须 scan_parquet:read_parquet().lazy() 是反模式。
  6. Parquet 不是万能:小数据 / upsert / OLTP 别用它。
  7. 90% 单机场景 → DuckDB:SQL + parquet + 零依赖。
