数据质量 —— Great Expectations PoC

L04 让你知道"数据长什么样"。本课回答"数据好不好" —— 用 GX 把"质量"从形容词变成可执行的断言。

节奏:约 75 分钟 · 产出物:可运行的 GX 期望套件 + DQ 报告
⚠ API 时效(2026-07)

本课基于 GX Core 1.x(2024 起 GA,2026-07 当前版本 1.19.0)。如果你看到的教程用 validator.expect_column_* 流式 API,那是 0.18 旧 API,已在 1.0 移除,会直接报错。本课全程用新的 class-based API:gx.expectations.ExpectColumnValuesToNotBeNull(...)。

  2. HTML DQ 报告(Data Docs) —— 自动生成的可分享报告

1. 数据质量的六个维度(10 分钟)

"数据质量好不好"是空话。DMBOK 和 ISO/IEC 25012 把它拆成可度量的维度。下面是 6 个最常用的:

维度含义典型规则
完整性 Completeness该有的不能空user_id IS NOT NULL
唯一性 Uniqueness不该重复的不重复user_id 唯一
有效性 Validity符合格式/范围age BETWEEN 18 AND 120
一致性 Consistency跨表/跨系统对得上订单总额 = 子项求和
时效性 Timeliness数据新鲜updated_at > now() - INTERVAL '1 day'
准确性 Accuracy与现实世界相符电话号码能拨通(最难,通常需外部验证)
关键认知

这六个维度里,前四个可以机器化检测(完整性/唯一性/有效性/一致性);时效性靠 SLA 监控;准确性最难,通常需要人工或外部源校验。工具能解决的是前五个 —— 准确性必须靠业务流程。不要指望 GX 解决"数据准不准"问题。

Q1 · 下面哪个数据质量维度最难用工具自动检测?
  • 完整性(非空)
  • 准确性(与现实相符)
  • 唯一性(不重复)
  • 有效性(格式/范围)
答案:B。完整性(A)、唯一性(C)、有效性(D)都是结构性检查,工具直接能跑。准确性(B)需要与现实世界对照(电话能拨通吗、地址真实吗、收入对吗),必须人工抽样或对接外部权威源。这是为什么数据治理项目通常先解决前 4 维,准确性留给业务流程。这也是为什么"数据准不准"不能只靠 DQ 工具回答。

3. GX Core 1.x 的对象图(10 分钟)

理解 GX 1.x 的对象关系,后面写代码才不会迷路。一张图:

DataContext                        # 入口,mode="file" 才能生成 Data Docs
 └─ data_sources.add_pandas(...)   # 数据源(Pandas/SQL/Spark)
     └─ add_dataframe_asset(...)   # 数据资产
         └─ add_batch_definition_whole_dataframe(...)  # 批定义
                ↕ (ValidationDefinition 关联)
context.suites.add(ExpectationSuite)   # 期望套件
     └─ add_expectation(gx.expectations.ExpectColumnToExist(...))
                ↕
context.checkpoints.add(Checkpoint(validation_definitions=[...]))
                ↓ checkpoint.run(batch_parameters={"dataframe": df})
        Validation Results → Data Docs(HTML 报告)
3 个核心概念
  1. Expectation(期望):一条可执行的断言,如"age 列值在 18-120 之间"。这是数据契约的最小单位。
  2. ExpectationSuite(期望套件):一组 expectation,描述"一张表应该长什么样"。是数据契约的载体。
  3. Checkpoint:把套件 + 数据 + 触发动作绑一起,执行后产出 Validation Results。是DQ 任务的可执行单元。

4. 完整可运行示例(30 分钟)

下面这个脚本复制粘贴就能跑。它创建一份"好数据",定 6 条 expectation,然后用"坏数据"跑校验,最后生成 HTML 报告。

"""gx_demo.py — GX Core 1.x 完整 PoC(2026-07 验证可跑)"""
import great_expectations as gx
import pandas as pd

# --- 1. 创建 FileDataContext(必须 file 模式才能生成 Data Docs)---------
context = gx.get_context(mode="file", project_root_dir=".")

# --- 2. 构造"已知良好"参考数据 -----------------------------------------
good_df = pd.DataFrame({
    "user_id":   [1, 2, 3, 4, 5],
    "email":     ["a@x.com","b@x.com","c@x.com","d@x.com","e@x.com"],
    "age":       [25, 30, 35, 40, 22],
    "country":   ["US", "CA", "US", "GB", "US"],
})

# --- 3. 连接数据(Data Source → Asset → Batch Definition)-------------
data_source = context.data_sources.add_pandas("pandas_src")
data_asset  = data_source.add_dataframe_asset(name="users_asset")
batch_def   = data_asset.add_batch_definition_whole_dataframe("users_batch_def")

# --- 4. 创建期望套件 + 6 条 expectation -------------------------------
suite = context.suites.add(gx.ExpectationSuite(name="user_suite"))

# 完整性:user_id 不能空
suite.add_expectation(gx.expectations.ExpectColumnValuesToNotBeNull(column="user_id"))
# 存在性:列必须存在
suite.add_expectation(gx.expectations.ExpectColumnToExist(column="email"))
# 唯一性:user_id 不重复
suite.add_expectation(gx.expectations.ExpectColumnValuesToBeUnique(column="user_id"))
# 有效性:年龄范围
suite.add_expectation(gx.expectations.ExpectColumnValuesToBeBetween(
    column="age", min_value=18, max_value=120))
# 有效性:国家枚举
suite.add_expectation(gx.expectations.ExpectColumnValuesToBeInSet(
    column="country", value_set=["US", "CA", "GB"]))
# 有效性:邮箱格式
suite.add_expectation(gx.expectations.ExpectColumnValuesToMatchRegex(
    column="email", regex=r"^[^@]+@[^@]+\.[^@]+$"))

# --- 5. ValidationDefinition 把"数据"和"期望"绑起来 ------------------
validation_def = context.validation_definitions.add(
    gx.core.validation_definition.ValidationDefinition(
        name="users_validation",
        data=batch_def,
        suite=suite,
    )
)

# --- 6. Checkpoint 绑定执行单元 ---------------------------------------
checkpoint = context.checkpoints.add(
    gx.checkpoint.checkpoint.Checkpoint(
        name="users_checkpoint",
        validation_definitions=[validation_def],
    )
)

# --- 7. 跑"坏数据",看哪些 expectation 失败 ---------------------------
bad_df = good_df.copy()
bad_df.loc[1, "user_id"] = 1          # 重复 → 唯一性失败
bad_df.loc[2, "age"]     = 150        # 越界 → between 失败
bad_df.loc[3, "country"] = "XX"       # 不在枚举 → in_set 失败
bad_df.loc[4, "email"]   = "not-an-email"  # 格式错 → regex 失败

result = checkpoint.run(batch_parameters={"dataframe": bad_df})
print(result.describe())

# --- 8. 生成 HTML 报告 -------------------------------------------------
context.build_data_docs()
print(context.open_data_docs())  # 打开浏览器或打印 file:// URL
# 默认输出位置:./gx/uncommitted/data_docs/local_site/index.html

预期结果

脚本跑完后,终端输出会显示 4 条失败(唯一性/between/in_set/regex 各一条)。浏览器自动打开 Data Docs,你能看到每个 expectation 的通过/失败率 + 失败样本。

expectationgood_df 结果bad_df 结果
ExpectColumnToExist (email)✅✅
ExpectColumnValuesToNotBeNull (user_id)✅✅
ExpectColumnValuesToBeUnique (user_id)✅❌(1 个重复)
ExpectColumnValuesToBeBetween (age)✅❌(150 越界)
ExpectColumnValuesToBeInSet (country)✅❌(XX 非法)

Q2 · GX Core 1.x 中,一组 expectation 放在哪个对象里?
  • Checkpoint
  • Validator
  • ExpectationSuite
  • DataContext
答案:C。ExpectationSuite 是一组 expectation 的容器,描述"一张表应该长什么样" —— 即数据契约的载体。A(Checkpoint)是把 suite+数据+动作绑起来执行的单元;B(Validator)是 0.18 旧 API,1.x 已删除;D(DataContext)是入口,管理 suites/checkpoints/data_sources 全局。

5. GX vs Soda vs dbt tests(5 分钟)

三个工具解决同类问题,选型看场景:

维度GXSoda Coredbt tests
风格Python,类 API声明式 YAMLSQL + YAML
报告Data Docs HTML(最强)Soda Clouddbt docs
适合复杂校验、多源、需 Python声明式、非工程师友好已在 dbt 上的项目
学习曲线陡(API 多)平(YAML 简单)低(顺道)
本课选它原因最完整,适合教学——
实战建议

已在 dbt → 用 dbt tests(零成本,顺道覆盖);多源/复杂校验/需 HTML 报告 → GX;团队非工程师多 → Soda。三者可并存:dbt tests 守 SQL 模型,GX 守跨源契约,Soda 做轻量监控。不要为了用工具而用工具 —— L03 自评里 DQ 是短板,但 DQ 工具本身不解决问题,规则的设计和 Owner 的归属才解决。

6. 把 DQ 结果送进 DataHub(可选,带重要 caveat)

⚠ 时效风险(2026-07)

DataHub 的 DataHubValidationAction 官方测试范围是 GX 0.18.x,不确认支持 GX Core 1.x(详见 DataHub 集成文档)。本节作为概念介绍,实际落地前务必先验证你的 GX 版本与 plugin 兼容。如不兼容,改用 GX 自带 Data Docs + Slack/邮件告警也是合理路径。

理想流程:DQ 失败 → DataHub 显示该表"DQ 状态:failed"→ Steward 收到通知。这是 DMBOK "Metadata" 和 "Data Quality" 两个知识区在工程上的交汇。配置思路: 

# 安装 plugin(注意:可能仅兼容 GX 0.18)
pip install acryl-datahub-gx-plugin

# Checkpoint 配置里加 action(0.18 风格 YAML)
action_list:
  - name: datahub_action
    action:
      module_name: datahub_gx_plugin.action
      class_name: DataHubValidationAction
      server_url: http://localhost:8080  # L04 起的 DataHub

如果遇到版本不兼容报错,降级路径是:GX 跑出结果 → 用脚本推 DataHub REST API → 同样效果,只是少了一层自动化。

## 10 条 expectation

| # | expectation 类                          | 字段        | 参数                  | 维度    | 失败动作        |
|---:|----------------------------------------|------------|----------------------|---------|----------------|
|  1 | ExpectColumnToExist                    | user_id    | -                    | 完整性  | 阻断 + 告警     |

|  4 | ExpectColumnValuesToBeBetween          | age        | 18-120               | 有效性  | 告警            |
|  5 | ExpectColumnValuesToBeInSet            | country    | [US,CA,GB,CN,...]    | 有效性  | 告警            |
|  6 | ExpectColumnValuesToMatchRegex         | email      | ^[^@]+@[^@]+\.[^@]+$ | 有效性  | 告警            |
|  7 | ExpectColumnValuesToNotBeNull          | created_at | -                    | 完整性  | 阻断            |
|  8 | ExpectColumnValuesToBeBetween(max_ts)  | created_at | ≤ now()              | 有效性  | 告警(未来数据) |
|  9 | [自己加,如跨表一致性]                  |            |                      | 一致性  |                |

## 失败处理约定
- 阻断级:Pipeline 停,值班 Steward 接到告警 30 分钟内响应
- 告警级:进 DQ 看板,周会复盘
- 通知渠道:Slack #data-quality + 邮件

本课要点回顾

  1. 质量 = 可度量维度:完整性/唯一性/有效性/一致性可机器化,时效性靠 SLA,准确性靠业务。
  2. GX Core 1.x API:class-based gx.expectations.Expect*,旧 validator.expect_* 已废。
  3. 三对象:Expectation(断言)→ Suite(契约)→ Checkpoint(执行)。
  4. 包名 great_expectations,不是 gx,但 import 别名是 gx。
  5. Data Docs 必须 FileDataContext:ephemeral 模式跑校验但不能出报告。
  6. 工具是手段不是目的:规则设计和 Owner 归属比工具选型更重要。
