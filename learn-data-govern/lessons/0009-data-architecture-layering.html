承接关系

L04(目录)/ L05(质量)/ L06(血缘)都在表已存在的前提下讲治理。但表怎么分层是 L01–L08 一直悬而未决的底层问题。本课补上:它对应 DCMM "数据架构"能力域,是 L04-06 的物理基础。学完你会突然明白为什么 L06 血缘图天然是"漏斗形"。

本课产出物

  2. 每层治理矩阵 —— 把 L04-06 的动作映射到不同层
  3. 可运行 dbt + DuckDB 项目 —— 电商四层最小工程

1. 为什么必须分层(8 分钟)

不分层的代价

想象一个公司所有数据都在一个 schema 里,业务库的 orders 表直接给 BI 查。会发生什么?

问题具体表现
口径冲突BI 团队 A 算"GMV"含退款,B 团队 B 不含,两张报表永远对不上(L01 章程里的经典痛点)
重复计算10 个看板各自 JOIN 同样的 users × orders,跑 10 次相同逻辑
改不动业务库 schema 变了,所有下游报表/ETL 一起爆,改一次伤筋动骨
权限混乱分析师直接查业务库 = 全公司都能看原始数据(L07 合规风险)
血缘难解没有清晰的"哪一层给谁消费",L06 血缘图变成一团乱麻
分层的核心价值

分层 = 用空间换秩序:每一层只解决一类问题,层间通过清晰契约(表名约定 + 字段口径 + 物化方式)耦合。源系统变了,只影响 ODS 层;BI 改需求,只在 ADS 层动。这是 L06 血缘能画出"漏斗形"的根本原因。

2. 四层架构 ODS / DWD / DWS / ADS(20 分钟)

国内主流约定(源自阿里 OneData 体系,DataWorks 普及)。四层各有清晰责任:

层全称责任表名约定粒度
ODSOperational Data Store贴源、1:1 镜像、轻清洗ods_xxx 或 stg_xxx同业务库
DWDData Warehouse Detail明细事实、维度退化、去重、扩字段dwd_xxx_di(di=detail)1 行/业务事件
DWSData Warehouse Service轻度汇总,服务多场景dws_xxx_da(da=day)1 行/(实体,周期)
ADSApplication Data Service面向应用/BI 的最终指标ads_xxx1 行/应用对象

每层"做什么 / 不做什么"

层✅ 该做❌ 不该做
ODS同步源数据、字段重命名、类型转换、简单标准化JOIN、聚合、业务逻辑
DWD去重、退化维度、关联基础维表、补字段跨业务域 JOIN、汇总
DWS按主题轻度汇总(user/day、product/week)面向单一应用做特化
ADS最终指标、宽表、面向 BI/API被下游再汇总(它就是终点)

命名后缀速记(阿里巴巴约定,国内主流): _di(detail)/ _da(day agg)/ _dm(month agg)/ _df(full snapshot)/ _i(incremental)。

漏斗形数据流

ODS 表最多(源系统镜像)→ DWD 收敛(去重 + 业务粒度)→ DWS 进一步收敛(汇总)→ ADS 最少(宽表)。表数量呈漏斗递减。这就是 L06 血缘图天然像漏斗的原因 —— 不是巧合,是分层架构的必然形态。

Q1 · 业务方要在报表里算"GMV",这个"GMV"字段应该在哪一层定义?
  • ODS 层,贴近源头
  • DWD 层,明细事实
  • ADS 层,面向应用
  • 直接在业务库加字段
答案:C。GMV 是面向应用的最终指标(ADS 职责)。DWD 只放明细(每笔订单的 amount),不预先聚合;ADS 才把 DWS 的日汇总进一步聚合成"GMV"。放 ODS(A)是越层;放 DWD(B)是混了职责(DWD 不该做应用特化);放业务库(D)是反模式(业务库 schema 不该为分析改)。这就是为什么"一张表的指标"必须有明确归属层。

3. 维度建模:DWD/DWS 的灵魂(15 分钟)

Kimball 方法

DWD/DWS 用的是 Kimball 维度建模。核心是事实表(fact)+ 维度表(dimension)的星型结构。比 Inmon 的企业数仓更轻,适合现代数据栈。

概念定义例子
事实 Fact可度量的业务事件一笔订单、一次点击、一次转账
维度 Dimension描述事实的上下文用户、商品、时间、地理
粒度 Grain一行事实代表什么"一行 = 一笔已支付订单"
度量 Measure事实里的数值字段amount、quantity、duration

缓慢变化维(SCD)

维度会变化(用户改了手机号、商品改了类目)。怎么记录历史?

类型策略用在哪
SCD1直接覆盖旧值不关心历史(如用户的临时昵称)
SCD2新增一行 + 起止时间 + 是否当前需追溯历史(用户曾经归属哪个销售)
SCD3加"上一个值"列只关心"上一次变化"
为什么 SCD2 是 DWD 的核心

没做 SCD2 的 DWD = 报表永远只能看"现在",不能回答"3 个月前这个订单属于哪个类目"。SCD2 = 给 DWD 装上时间机器。在 dbt 里用 snapshot 机制实现,这是 L03 数据生存周期知识区的工程对应。

Q2 · 用户去年改过一次手机号,业务方要"按历史手机号归属分析订单"。维度表应该用?
  • SCD1,直接覆盖
  • SCD2,新增行 + 时间段
  • SCD3,加"上一手机号"列
  • 不存历史,实时查业务库
答案:B。需要追溯完整历史 → SCD2(每个版本一行,带 valid_from/valid_to)。A(SCD1)丢历史;C(SCD3)只存"上一个",改两次以上就丢中间;D 是反模式(每次查业务库性能灾难 + 业务库可能已覆盖)。SCD2 是 DWD 的核心模式之一。

4. 每层治理动作差异(承接 L04-06)(15 分钟)

这是本课最关键的承接表 —— 把 L04(目录)/L05(质量)/L06(血缘)的动作按层差异化。

动作ODSDWDDWSADS
DQ 严格度 最严:不允许空/重复/格式错 严:粒度一致性 + 外键完整 中:量纲正确、SLA 时效 松:业务侧复核即可
失败处理(L05) 阻断 + 强告警 阻断 告警 + 降级 告警
血缘(L06) 源 → ODS 一对一 ODS → DWD 多对少 DWD → DWS 收敛 DWS → ADS 最收敛
目录字段(L04) 技术元数据为主 业务定义必填 口径说明 + Owner 对外 API 文档
SLA 分钟级(贴源) 小时级 日级 日级/按需
权限 仅 ETL(绝不对业务开放) 数据团队 分析师 业务方 + BI
⚠ 最常见的越层事故

"业务方直接查 ODS" —— 这是头号违规。ODS 是源数据的镜像,字段语义未规范化、口径未对齐、无 DQ 保证。业务只应该查 ADS,至多 DWS。一旦让业务养成查 ODS 的习惯,L04 目录/L07 合规就全废了。权限是分层能落地的最后防线。

5. dbt + DuckDB 实操:电商四层(20 分钟)

⚠ 技术栈说明(2026-07)

用 dbt Core 1.11 + dbt-duckdb 1.10(2026-05 / 2026-02 最新稳定)。DuckDB 是本地嵌入式 OLAP,无需 Docker,无需 server,一个文件就是数据库。与 L04-06 的重型栈(DataHub/GX/Marquez)形成对比 —— 本课零依赖、5 分钟跑通。

第 1 步:环境(2 分钟)

# 用 venv 隔离(和 L04/L05 的 venv 分开)
python3 -m venv ~/dbt-venv
source ~/dbt-venv/bin/activate

# dbt Core 1.11 + duckdb adapter
pip install dbt-core dbt-duckdb

# 验证(Python 3.10+ 必需)
dbt --version

坑:如果用 pipx install dbt-core,加 adapter 要用 pipx inject dbt-core dbt-duckdb,不能 pipx install dbt-duckdb(会装到另一个隔离 env,dbt 找不到)。

第 2 步:初始化项目(3 分钟)

mkdir dw_lesson && cd dw_lesson
dbt init dw_lesson
# 交互式选择:选 duckdb,数据库名 ./dw.duckdb

初始化后的目录结构:

dw_lesson/
├── dbt_project.yml
├── models/
│   └── example/            # 删掉,我们建自己的
├── seeds/                  # 放 CSV 源数据
├── tests/
└── target/

第 3 步:配置分层(改 dbt_project.yml)

# dbt_project.yml(2026 写法,config-version 不再必填)
name: dw_lesson
version: 1.0.0
profile: dw_lesson

models:
  dw_lesson:
    staging: { +materialized: view,  +tags: ods  }   # ODS 层
    dwd:     { +materialized: table,  +tags: dwd }   # DWD 层
    dws:     { +materialized: table,  +tags: dws }   # DWS 层
    ads:     { +materialized: table,  +tags: ads }   # ADS 层

第 4 步:源数据(seed)

在 seeds/raw_orders.csv 放电商原始订单数据:

order_id,user_id,order_ts,status,amount
1001,501,2026-07-01 10:00:00,paid,99.50
1002,502,2026-07-01 11:30:00,paid,158.00
1003,501,2026-07-02 09:15:00,cancelled,99.50
1004,503,2026-07-02 14:00:00,paid,320.00
1005,502,2026-07-03 16:45:00,paid,75.00
dbt seed                # 把 CSV 灌进 DuckDB

第 5 步:声明源(ODS 入口)

# models/staging/_sources.yml
version: 2

sources:
  - name: raw
    schema: main
    description: "源数据(seed 灌入)"
    tables:
      - name: raw_orders
        columns:
          - name: order_id
            tests: [not_null, unique]
          - name: user_id
            tests: [not_null]

第 6 步:ODS 层 SQL(轻清洗)

-- models/staging/stg_orders.sql
{{ config(materialized='view', tags=['ods']) }}

select
    cast(order_id   as bigint)    as order_id,
    cast(user_id    as bigint)    as user_id,
    cast(order_ts   as timestamp) as order_ts,
    lower(status)                 as status,            -- 标准化
    cast(amount     as decimal(12,2)) as amount
from {{ source('raw', 'raw_orders') }}

第 7 步:DWD 层 SQL(明细事实)

-- models/dwd/dwd_orders_di.sql
{{ config(materialized='table', tags=['dwd']) }}

with stg as (
    select * from {{ ref('stg_orders') }}
),
-- 去重:保留每个 order_id 最新一行
deduped as (
    select * from (
        select *,
            row_number() over (
                partition by order_id order by order_ts desc
            ) as rn
        from stg
    ) where rn = 1
)
select
    order_id,
    user_id,
    cast(order_ts as date) as order_date,    -- 退化时间维
    status,
    amount
from deduped

第 8 步:DWS 层 SQL(轻度汇总)

-- models/dws/dws_user_da.sql
{{ config(materialized='table', tags=['dws']) }}

select
    user_id,
    cast(order_ts as date) as order_date,
    count(*)               as order_cnt,
    sum(amount)            as daily_gmv,
    count(distinct order_id) as distinct_orders
from {{ ref('dwd_orders_di') }}
group by 1, 2

第 9 步:ADS 层 SQL(应用指标)

-- models/ads/ads_user_metrics.sql
{{ config(materialized='table', tags=['ads']) }}

select
    user_id,
    sum(order_cnt)      as total_orders,
    sum(daily_gmv)      as lifetime_gmv,
    min(order_date)     as first_order_date,
    max(order_date)     as last_order_date
from {{ ref('dws_user_da') }}
group by 1

第 10 步:跑 + 看血缘

dbt run                              # 按依赖顺序跑所有层
dbt run --select tag:dwd+            # 只跑 DWD 及下游
dbt docs generate && dbt docs serve  # 浏览器 http://localhost:8080 看血缘

打开 dbt docs,你会看到完美的漏斗形血缘图:1 个 source → 1 个 ODS → 1 个 DWD → 1 个 DWS → 1 个 ADS。这就是 L06 讲的"血缘 DAG 天然漏斗"的物理来源。

6. 常见反模式(5 分钟)

反模式为什么错正确做法
ODS 直接给 BI无 DQ 保证、口径未对齐、权限混乱只让 BI 查 ADS
DWS 加业务逻辑通用汇总层被单一应用绑架业务逻辑下移到 ADS
跨层依赖ADS 直接 ref ODS,绕过 DWD/DWS每层只能 ref 上一层
ADS 被 ADS 依赖ADS 是终点,不该被汇总把公共部分提到 DWS
所有层 materialized=tableODS 用 view 更灵活、省存储ODS view,DWD+ table
Q3 · 你的分析师说"ODS 太慢,我直接连源 MySQL 查"。最合适的回应?
  • 同意,让他用只读账号直连
  • 拒绝,让他改查 ADS
  • 让他查 DWD
  • 把 ODS 物化成 table 加速
答案:B。业务方只应该查 ADS,至多 DWS。直连源 MySQL = 越层 + 权限混乱 + 业务库被分析查询拖垮。ODS 慢的解决之道是把 ADS 做好,而不是放行业绕过架构。A 违反 L07 合规(业务库直接暴露);C 越层(DWD 不直接给业务);D 治标不治本(ODS 物化加速仍不解决"业务该不该看 ODS")。

本课要点回顾

  1. 分层 = 用空间换秩序:每层只做一类事,层间靠清晰契约耦合。
  2. 四层职责:ODS 贴源 / DWD 明细事实 / DWS 轻度汇总 / ADS 应用指标。
  3. Kimball 维度建模:事实表 + 维度表 + 粒度 + SCD2(历史追溯)。
  4. 每层治理差异:DQ 严格度、SLA、权限从 ODS→ADS 递减;血缘天然漏斗形。
  5. dbt + DuckDB 是 2026 最轻量落地栈:零 Docker,5 分钟跑通四层。
  6. 越层是头号违规:业务只查 ADS,权限是最后防线。
