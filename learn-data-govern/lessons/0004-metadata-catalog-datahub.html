元数据与目录 —— DataHub PoC

DCMM 自评告诉你"数据标准"是短板。本课用开源工具把短板补起来 —— 在本地起一个可用的数据目录,采集真实元数据。

节奏:约 90 分钟(含 Docker 拉镜像等待)· 产出物:本地 DataHub 实例 + 一份元数据采集 recipe
⚠ 本课硬件要求

DataHub quickstart 起 ~10 个容器(MySQL + Kafka + Zookeeper + Elasticsearch + GMS + frontend 等),最低 8GB 空闲内存,推荐 16GB。Elasticsearch 是最常 OOM 的容器。提前关掉其他重型应用。

本课产出物
  1. 本地 DataHub 实例 —— 跑在 http://localhost:9002
  2. demo-data recipe.yml —— 采集样例元数据的配置文件
  3. 自己场景的元数据采集计划 —— 列出真实要接的 3 个数据源

1. 数据目录到底解决什么(10 分钟)

"数据目录"听起来像 90 年代的"数据字典",但两者有本质差异。理解这个差异,你才知道为什么 DCMM 把"数据标准"单独列成一个能力域。

维度数据字典(老)数据目录(新)
载体Excel / Word 文档可交互的服务 + API
更新人工维护,常过期自动从源系统采集
覆盖表名 + 字段注释schema + 血缘 + Owner + 术语 + 质量 + 使用统计
用法打开文档查搜索、订阅变更、API 集成
谁负责某个 DBA/文档员分布式 Steward 网络
目录 = 元数据的中央枢纽

目录的核心价值不是"存元数据",是"让元数据流动起来":质量工具把 DQ 结果写进来、调度工具把血缘写进来、Steward 把业务定义写进来、消费者通过目录找到数据。目录是 DMBOK "Metadata" 知识区的工程实现。

开源目录横评(2026)

工具起源特点本课选不选
DataHubLinkedIn / Acryl功能最全,血缘/术语/Owner 强✅ 本课主选
OpenMetadataCollateUI 紧凑、内置 DQ + SLOL05 备选
AmundsenLyft轻量、搜索优先,血缘弱✗
MarquezWeWork专注 OpenLineage,不做完整目录L06 主选
Apache AtlasHortonworksHadoop 系,发版慢(2025-01)✗ 不推荐 2026 greenfield
Q1 · 数据目录与数据字典最本质的差异是?
  • 目录用网页,字典用文档
  • 目录自动采集并流通元数据
  • 目录收费,字典免费
  • 目录字段多,字典字段少
答案:B。载体(网页 vs 文档)只是表象,本质是元数据的流动方式:字典靠人工维护(必然过期),目录靠自动采集 + API 流通(实时)。目录让质量、血缘、Owner、业务定义、使用统计汇聚到一处并互相联动,这是字典做不到的。A 是表象;C 错(都有开源);D 是结果不是根因。

2. 在本地起 DataHub(20 分钟 + 拉镜像时间)

第 1 步:确认 Docker 就绪(2 分钟)

# 1. 确认 docker 命令不需要 sudo(关键!)
docker info >/dev/null 2>&1 && echo "✅ docker ok" || echo "❌ docker 需要 sudo 或没运行"

# 如果上面报错,先把用户加入 docker 组:
# sudo usermod -aG docker $USER && newgrp docker

# 2. 确认 compose v2
docker compose version
⚠ sudo 是头号杀手

DataHub CLI 调 docker 命令时不传 sudo。如果你的 docker 需要 sudo 才能跑,quickstart 会失败报"Docker doesn't seem to be running"。先解决用户组问题,再继续。

第 2 步:安装 DataHub CLI(3 分钟)

# 强烈建议用 venv —— DataHub 依赖 pydantic 1.x,会和系统其他包冲突
python3 -m venv ~/datahub-venv
source ~/datahub-venv/bin/activate

# 注意:PyPI 上叫 acryl-datahub,不是 datahub(datahub 是另一个无关包!)
pip install acryl-datahub

# 验证
datahub version

坑:包名是 acryl-datahub,但安装后命令是 datahub。很多博客写 pip install datahub —— 那是另一个完全无关的项目。

第 3 步:启动整套服务(5 分钟 + 拉镜像)

# 一条命令拉起所有服务(MySQL/Kafka/ES/GMS/Frontend/...约 10 个容器)
datahub docker quickstart

# 首次会拉几个 GB 镜像,耐心等。后续启动快。
# 启动过程中另开一个终端窗口查看状态:
datahub docker info

第 4 步:访问 UI(2 分钟)

项值
URLhttp://localhost:9002
用户名datahub
密码datahub
REST API(采集元数据用)http://localhost:8080

登录后先看到空搜索结果(正常,还没采集任何元数据)。下一步灌入数据。

3. 采集第一份元数据(20 分钟)

用 demo-data 源(最简,零外部依赖)

demo-data 是 DataHub 内置的样例数据包,包含几十张表、几个 dashboard、术语表节点、示例用户、血缘边。零外部依赖,用来演示完整流程最合适。 

# 创建 recipe 文件
cat > demo_recipe.yml <<'EOF'
source:
  type: demo-data
  config:
    pack_name: bootstrap

sink:
  type: datahub-rest
  config:
    server: http://localhost:8080
EOF

# 执行采集
datahub ingest -c demo_recipe.yml

预期输出末尾:

... FINISHED: ...
- Wrote 86 Events
- Coverage: 100.00%
- Connectors ... 1
- Sources ... 1

看到 "100.00% Coverage" 即成功。回 UI 刷新,看到几十张数据集。

UI 三大功能走一遍(在浏览器里点)

功能在哪看什么
数据集 + Schema 顶部 Search → 筛选 Dataset → 打开任一表 字段名、类型、描述、Owner
血缘 Lineage 表详情页 → Lineage 标签 上下游表的 DAG(本课高光时刻)
术语表 Glossary 左侧 Govern → Glossary 业务术语定义,可挂到字段上
Owner 设置 表详情页 → Owners 为表指定 Steward/Owner
Q2 · 在 DataHub 中,采集元数据的配置文件叫什么?
  • Connector
  • Pipeline
  • Recipe
  • Manifest
答案:C。DataHub 把 source + sink 配置叫 Recipe(YAML 格式),用 datahub ingest -c recipe.yml 执行。A 是 source 类型(如 demo-data、postgres、snowflake 都是 connector);B 是通用词不是 DataHub 术语;D 是 dbt 的概念(完全不同的工具链)。

4. 接真实数据源(15 分钟)

demo-data 是演示用的。真实场景你要接的是 Postgres / MySQL / dbt / Kafka / Snowflake 等真实源。这里以 Postgres 为例(很多公司有 Postgres),其他源模式一样。 

# 1. 安装 postgres connector 扩展
pip install 'acryl-datahub[postgres]'

# 2. 写真实 recipe
cat > postgres_recipe.yml <<'EOF'
source:
  type: postgres
  config:
    host_port: your-db-host:5432
    database: your_db
    username: readonly_user
    password: ${PG_PASSWORD}      # 从环境变量取,不要写明文
    # 可选:只采某些 schema
    schema_pattern:
      allow: ["public", "analytics"]

sink:
  type: datahub-rest
  config:
    server: http://localhost:8080
EOF

# 3. 设置密码环境变量后执行
export PG_PASSWORD='...'
datahub ingest -c postgres_recipe.yml
⚠ 元数据采集 ≠ 数据访问

采集元数据只需要只读权限到 information_schema。绝不要用读写账号!给采集账号单独建一个 readonly_user,只授予 USAGE + SELECT on information_schema。目录不应该有读取真实数据的能力 —— 这是 L07 合规课会强调的安全边界。

新建 deliverables/catalog-source-plan-v0.1.md。基于你 L01 钉死的项目场景,列出真实要接的 3 个数据源:

# 数据目录采集计划 v0.1

## 优先采集的 3 个源
| 序 | 源类型         | 主机/位置         | 覆盖业务域      | 采集账号      | 优先级 |
|---:|---------------|------------------|----------------|--------------|:-----:|
|  1 | postgres      | prod-db:5432     | 用户画像        | ro_catalog   | P0    |
|  2 | dbt           | dbt cloud / 本地  | 数仓模型        | api_token    | P0    |
|  3 | kafka         | kafka-broker:9092 | 行为事件        | ro_consumer  | P1    |

## 决策依据
1. 为什么是这 3 个:[L01 场景里最痛的"口径不一致"在这 3 个源里发生]
2. 为什么不是别的:[业务侧 BI 看板暂不接入,因为上层依赖 P0 三源完成]

## 风险/前置
- [PG 只读账号需要 DBA 审批,预计 1 周]
- [dbt cloud token 需要数据团队 owner 授权]

## 上线节奏
- W1: demo-data 跑通,Steward 培训

5. 常见问题与排障(5 分钟)

症状原因解决
datahub docker quickstart 报 "Docker doesn't seem to be running" docker 需要 sudo sudo usermod -aG docker $USER && newgrp docker
Elasticsearch 容器反复重启 内存不足(8GB 是下限) 关掉其他重型应用;或给 Docker 分配 ≥12GB
UI 9002 打不开但 docker ps 显示 frontend 在跑 GMS 还在初始化 等 datahub docker info 显示 GMS healthy,通常 3-8 分钟
ingest 报版本不匹配 CLI 与镜像版本漂移 pip install --upgrade acryl-datahub
pydantic 错误 系统 Python 装了 pydantic 2 必须用 venv(已强调);pip install pydantic==1.10.x

停服与清理

# 临时停(保留数据卷,下次启动恢复)
datahub docker quickstart --stop

# 彻底清空(删容器 + 数据卷 + ES 索引,回到初始状态)
datahub docker nuke
# 慎用!会丢失所有已采集的元数据
Q3 · 元数据采集账号应该用什么权限?
  • DBA 全权限,方便诊断
  • 只读 information_schema
  • 读写业务表权限
  • 不需要账号,匿名即可
答案:B。目录只需读 information_schema(表/列/类型/注释),不应该有读真实数据的能力。这是合规边界(L07 课会详讲):一旦目录账号能 SELECT 真实业务数据,目录本身就成了攻击面。A 是过度授权(灾难);C 错(目录不需要看真实数据);D 不可能(任何 DB 都要账号)。原则:工具按需授权,最小权限。

本课要点回顾

  1. 数据目录 ≠ 数据字典:目录的核心是元数据流动(自动采集 + API 集成),字典是静态文档。
  2. DataHub 是 2026 默认选型:Apache Atlas 节奏慢不推荐 greenfield。
  3. 包名 acryl-datahub,命令 datahub:不要装错包。
  4. 用 venv 隔离:DataHub 依赖 pydantic 1.x,会和系统包冲突。
  5. 采集账号最小权限:只读 information_schema,不能碰真实数据。
  6. Recipe 是核心抽象:source + sink 配置,一行命令跑采集。
