1. 血缘解决什么(10 分钟)

血缘的价值 —— 三个真实场景

场景没有血缘时有血缘时
影响分析 改这张表的字段,不知道下游哪些报表会爆 看血缘 DAG,秒级列出所有下游消费者
根因定位 报表数字异常,挨个查上游表 沿血缘反向追溯,直奔源头
合规审计 无法证明"这份用户数据经历了哪些加工、流向何处" 血缘 + 操作元数据 = 审计所需轨迹
血缘的本质

血缘不是"画图好看",是可追溯性(Traceability)的工程化。它把分散在 ETL、调度器、SQL、表注释里的"数据流向信息"自动采集到一处。这才是 DCMM "数据架构"能力域的核心要求 —— 有图,有据,有更新。

血缘的三个层次

层粒度典型工具/方式
表级 Table-level表 A → 表 B调度器元数据、手工标注
列级 Column-level列 X 来自列 YSQL 解析(如 dbt、Spark 解析器)
操作级 Operational哪个 job 在何时跑了,状态如何OpenLineage 事件流

层级取舍:表级最简单,通常先做(满足 80% 场景);列级最有用但最难(SQL 解析易错);操作级解决"哪天哪个版本出错"。OpenLineage 三层都覆盖,这就是它成为标准的根本原因。

Q1 · 数据血缘最直接的价值场景是?
  • 让看板更美观
  • 影响分析与根因定位
  • 减少存储成本
  • 加快查询速度
答案:B。血缘的核心价值是可追溯性:改字段前做影响分析(下游谁会爆)、出问题时做根因定位(数字从哪来错了)。A(美观)是副作用;C/D 与血缘无关(那是存储/查询优化的事)。血缘本质是关于数据流向的知识,不是性能工具。

2. OpenLineage 标准(10 分钟)

OpenLineage 是开放血缘采集标准(LF AI & Data 基金会托管)。它的核心创新是定义了统一的血缘事件格式,让 Airflow / Spark / dbt / Flink / Snowflake 等不同工具用同一种方式报告"我做了什么、动了哪些数据"。

OpenLineage 事件的核心字段

{
  "eventType": "START | RUNNING | COMPLETE | FAIL | ABORT",
  "job": {
    "namespace": "my_pipeline",
    "name": "etl_users_daily"
  },
  "run": {
    "runId": "uuid-...",
    "startedAt": ...,
    "endedAt": ...
  },
  "inputs":  [ {"namespace":"pg","name":"raw.users"} ],   # 上游
  "outputs": [ {"namespace":"pg","name":"dw.dim_users"} ] # 下游
}
字段含义
job谁做的(job 名 + namespace)
run具体某次执行(runId + 时间)
inputs读了哪些数据集(上游)
outputs写了哪些数据集(下游)
eventType事件状态(START/RUNNING/COMPLETE/FAIL)

Marquez 是参考实现

Marquez 是 OpenLineage 的参考后端(WeWork 起源,Apache 2.0):它接收 OpenLineage 事件、存储、提供 Web UI + REST API。本课用它跑通端到端。

OpenLineage vs Marquez关系
OpenLineage协议(事件格式 + 传输)
Marquez后端实现(接收 + 存储 + UI)
DataHub / OpenMetadata也支持 OpenLineage(但功能更广)
为什么先学 Marquez 不直接学 DataHub 接 OpenLineage

Marquez 专门做血缘,UI 聚焦、轻量、上手快;DataHub 是完整目录,血缘只是它一个标签。学血缘概念,用 Marquez 更清晰;落地到生产,通常会回到 DataHub / OpenMetadata(它们都支持 OpenLineage 协议,迁移成本低)。

4. 用 Python 发送血缘事件(25 分钟)

下面这个脚本不需要任何 ETL 框架,直接用 openlineage-python 把一个"ETL 任务"的血缘事件发到 Marquez。这让你理解 OpenLineage 的协议本身,而不是某个框架的封装。 

# 安装
python3 -m venv ~/lineage-venv && source ~/lineage-venv/bin/activate
pip install openlineage-python
"""lineage_demo.py — 手动发 OpenLineage 事件到 Marquez(2026 验证)"""
from openlineage.client import OpenLineageClient
from openlineage.client.run import (
    Run, Job, RunEvent, RunState,
    Dataset, InputDataset, OutputDataset,
)
import uuid
from datetime import datetime, timezone

# 1. 连到 Marquez
client = OpenLineageClient(url="http://localhost:5000")

# 2. 定义一个 ETL 任务:从 raw.users 加工到 dw.dim_users
run_id = str(uuid.uuid4())
job = Job(namespace="my_pipeline", name="etl_users_daily")

inputs = [InputDataset(
    namespace="postgres://prod",
    name="public.raw_users",
)]
outputs = [OutputDataset(
    namespace="postgres://prod",
    name="public.dim_users",
)]

# 3. 发 START 事件(任务开始)
client.emit(RunEvent(
    eventType=RunState.START,
    eventTime=datetime.now(timezone.utc).isoformat(),
    run=Run(runId=run_id),
    job=job,
    inputs=inputs,
    outputs=outputs,
    producer="learn-data-govern/lineage_demo",
))

# 4. ... 实际 ETL 在这里跑(读 raw_users,写 dim_users)...

# 5. 发 COMPLETE 事件(任务完成)
client.emit(RunEvent(
    eventType=RunState.COMPLETE,
    eventTime=datetime.now(timezone.utc).isoformat(),
    run=Run(runId=run_id),
    job=job,
    inputs=inputs,
    outputs=outputs,
    producer="learn-data-govern/lineage_demo",
))

print("✅ 血缘事件已发送,回 http://localhost:3000 查看")

跑完后回 Marquez UI,你会看到:

  • 左栏多了一个 namespace my_pipeline
  • 里面有 job etl_users_daily
  • 点开能看到 input: raw_users → output: dim_users 的血缘图
  • Runs 标签看到 START 和 COMPLETE 两次事件

扩展:多步 ETL 形成链

把脚本复制三份,改成不同的 job 名,让它们形成链:raw_users → dim_users → mart_user_metrics → bi_dashboard。每跑一次,Marquez UI 上的血缘 DAG 就延长一段。这就是真实数仓的血缘积累过程。

Q2 · OpenLineage 与 Marquez 的关系是?
  • 两者是同一工具的不同名称
  • Marquez 是协议,OpenLineage 是实现
  • OpenLineage 是协议,Marquez 是实现
  • 两者无关系,纯巧合命名
答案:C。OpenLineage 是协议/标准(定义事件格式);Marquez 是这个协议的参考实现(接收存储 + UI)。这是开放生态常见模式:标准先行(LF AI & Data 托管),实现可换(Marquez / DataHub / OpenMetadata 都支持 OpenLineage 协议)。B 反了;A 是把协议和实现混为一谈;D 完全错(两者紧密关联)。

5. 真实场景:框架集成(10 分钟)

生产中你不会手写事件,而是用框架封装好的 OpenLineage 集成:

框架集成方式采集粒度
Airflow装 airflow-openlineage-backend,自动采集所有任务表级 + 操作级
dbt配置 env: OPENLINEAGE_URL,dbt 自动上报表级 + 列级(SQL 解析)
Spark加 spark-openlineage listener表级 + 操作级
Snowflake原生支持(配置即可)表级 + 列级
FlinkFlink OpenLineage connector表级 + 操作级
推荐落地顺序

dbt → Spark → Airflow。原因:dbt 的 SQL 解析能给到列级血缘(最值钱),且配置最简单(env var);Spark 覆盖大数据处理;Airflow 最后,因为它不直接知道数据流向,只能靠 operators 解析(粒度粗)。如果你的 ETL 主要在 dbt,先接 dbt 收益最大。

# 血缘采集计划 v0.1

## 接入优先级

| 序 | job 名                  | 框架      | 主要 input → output              | 集成方式                  | 优先级 |
|---:|------------------------|----------|----------------------------------|--------------------------|:-----:|
|  1 | dbt: fct_user_metrics  | dbt      | dim_users → fct_user_metrics     | OPENLINEAGE_URL env var  | P0    |

Q3 · 列级血缘比表级血缘难,主要原因是?
  • 字段太多存储不下
  • 需解析 SQL 推导列依赖
  • UI 画不出来
  • 协议不支持
答案:B。列级血缘需要解析 SQL,理解 SELECT/JOIN/WHERE 中列与列的依赖关系(如 SELECT a+b AS c → c 依赖 a 和 b)。SQL 解析易错(子查询、CTE、UDF、动态 SQL),这是技术难点。A 错(字段多不是问题);C 错(UI 能画);D 错(OpenLineage 支持 datasets 的 facets 可表达列级)。这就是为什么 dbt(有完整 SQL AST)能提供高质量列级血缘,而 Airflow(只知任务名)只能给表级。

本课要点回顾

  1. 血缘 = 可追溯性的工程化:影响分析、根因定位、合规审计三大场景。
  2. 三个层次:表级(简单,80% 场景)/ 列级(最值钱,需 SQL 解析)/ 操作级(谁在何时跑了)。
  3. OpenLineage 是协议,Marquez 是参考实现:DataHub / OpenMetadata 也支持该协议。
  4. 事件五要素:job / run / inputs / outputs / eventType。
  5. 框架集成比手写好:dbt 最易接入(给列级血缘),Airflow 最粗(只给表级)。
  6. 落地顺序:dbt → Spark → Airflow:收益从高到低。
