量化数据层架构选型(A股散户)

前 8 课都是"假设数据已就绪"做数学。本课补工程短板:数据从哪来、怎么存、怎么查 —— 无 tick、散户/小团队视角。

节奏:约 80 分钟 · 产出物:数据层架构图 + 可运行的 AKShare→parquet 流水线 + 选型决策表
为什么量化要单独讲数据层

数据治理 L09(分层)+ L10(引擎选型)的框架对量化部分适用但不完全。量化的主轴是 symbol × timestamp(不是业务事件),且写少读多、因子列爆炸、点查与扫描并存。本课把这些差异讲透,并给散户可落地的方案。

本课产出物
  1. 数据层架构图(你的场景版):从 AKShare 到内存的完整链路
  2. 可运行的数据采集流水线:AKShare → 分区 parquet
  3. 选型决策表:什么场景换 ClickHouse / ArcticDB / KDB+
  4. 四个数据陷阱防护:前复权时变性 / 存活偏差 / 节假日 / 临时停牌

1. 量化和普通 OLAP 的本质差异(8 分钟)

维度普通 OLAP(L10)量化交易
主轴业务事件symbol × timestamp 双维度
写入随机用户写收盘批量灌入(每日 1 次)
读取聚合查询为主回测全市场扫描 + 点查并存
列数相对稳定因子爆炸(几十到上千列)
延迟秒级回测分钟级 + 实盘亚秒
形态表Tick / Bar / Factor / Fundamental / Alt 多形态
⚠ 这些差异决定选型不能照搬

L10 的结论"90% 单机场景 → DuckDB"在量化里仍然成立,但多了一个维度要权衡:时序与 symbol 的复合查询(如"找所有股票在过去 250 天的 MA20 最大值")。这种查询普通 OLAP 不常见,在量化里是日常。引擎的ASOF JOIN、rolling 窗口能力是关键。

3. 四个数据陷阱(必须先知道)(15 分钟)

这四个陷阱和数学无关,但会让你的回测结果系统性失真。新手最常踩,踩了不知道。

陷阱 1:前复权(qfq)有时变性

AKShare 的 stock_zh_a_hist(adjust="qfq") 给的是前复权价格。前复权锚定"今天的最新复权因子" —— 这意味着:

你今天拉的 2024 年 1 月某只股票的 qfq 收盘价,和下周该公司分红后你再拉的 2024 年 1 月 qfq 收盘价,会不一样。

后果:如果你跨日期拼接 qfq 数据(今天拉 2023,明天拉 2024),拼接处会出现幻觉跳空/涨跌停。这是回测失真的头号来源。

✅ 正确做法
  1. 存"不复权" + "复权因子"两份数据,按需现算 qfq/hfq
  2. 或用后复权(hfq):锚定 IPO 日,历史价格稳定,适合回测
  3. 绝不要在不同日期分批拉 qfq 然后拼接

陷阱 2:存活偏差(Survivorship Bias)

AKShare 的 stock_info_a_code_name 只返回当前在市的股票。如果你用它生成股票池,然后回测过去 5 年,你只测了"活下来的公司" —— 退市股(常常是表现差的)被系统排除,回测结果会高估策略表现。

✅ 正确做法
  1. 维护一份历史股票池(含已退市):Tushare stock_basic(list_status='D')(需积分)或定期快照累积
  2. 在 L05 数据质量检查里加"退市股覆盖率"作为度量

陷阱 3:交易日历(节假日)

回测必须用真实交易日,不是日历日。A 股一年约 242 个交易日(非 365)。国庆/春节闭市、临时休市(哀悼日)都要排除。

✅ 推荐

exchange_calendars 库(PyPI),免费无 token,包含 XSHG(上交所)日历,追踪临时休市。回测第一步: 

import exchange_calendars as xcals
sse = xcals.get_calendar("XSHG")
trading_days = sse.sessions_in_range("2019-01-01", "2024-12-31")

陷阱 4:除权除息日的收益断层

除权除息日(分红/送股),不复权价格会有跳空(看起来像暴跌),前复权/后复权会平滑掉。用哪个价格计算收益,直接决定回测结果。

你周一拉了 2023 年某股 qfq 数据,周三该公司分红后你又拉了 2023 年同股 qfq 数据。两次数据会?
  • 完全相同(历史不变)
  • 不同(qfq 锚定今天,分红后整段历史都变了)
  • 只有分红当天不同
  • 只有未来数据不同
答案:B。前复权锚定"今天"的最新复权因子 —— 公司分红后,复权因子变了,整段历史 qfq 价格都会被重新计算。这就是为什么"跨日期拼接 qfq" 会产生幻觉跳空。解决:存不复权+复权因子,现算 hfq 用于回测。A 错;C/D 是常见误解(以为只影响分红当天)。
用 AKShare 的 stock_info_a_code_name 生成股票池做 5 年回测,主要问题?
  • 数据有延迟
  • 列名不规范
  • 引入存活偏差(只含在市股票)
  • 退市股票数据过多
答案:C。stock_info_a_code_name 只返回当前在市股票,用它做历史回测 = 系统性排除了已退市(表现差的)公司 = 回测结果被高估。这是存活偏差的经典来源。解决:维护含退市股的历史股票池。A/B 是次要问题;D 反了(是缺退市股,不是多)。

关键设计:parquet 分区策略

方案问题是否推荐
按 date=YYYY-MM-DD 单层分区—✅ 推荐
按 symbol=AAPL 分区5000 symbol = 5000 个小文件 = 小文件灾难❌ 避免
按 year=/month=/day= 嵌套分区过多,元数据开销大❌ 避免
按 date + symbol bucket平衡文件数与并行度(分钟级数据用)✅ 分钟级推荐
⚠ 不要按 symbol 分区

5000 只 A 股 = 5000 个目录,每个目录一堆小 parquet。引擎扫描时元数据开销远超数据读取。正确:按 date 分区,row group 内按 symbol 排序(这样 parquet 的 row group 统计能对 symbol 做谓词下推)。

5. 可运行实操:AKShare → parquet 流水线(20 分钟)

# pip install akshare polars duckdb exchange_calendars pyarrow
import akshare as ak
import polars as pl
import exchange_calendars as xcals
from pathlib import Path
import time

DATA_DIR = Path("./data/daily")

def fetch_and_store(start="2019-01-01", end="2024-12-31"):
    # 1. 拿交易日历(避开节假日 + 临时休市)
    sse = xcals.get_calendar("XSHG")
    trade_days = [d.strftime("%Y%m%d") for d in
                  sse.sessions_in_range(start, end)]
    print(f"共 {len(trade_days)} 个交易日")

    # 2. 注意:这里用 stock_info_a_code_name 有存活偏差(见陷阱 2)
    #    生产应维护含退市股的历史股票池
    # 3. 按日期拉数据(不复权 + 复权因子分别存)
    for date_str in trade_days[-5:]:                    # 演示只跑最后 5 天
        try:
            # 拉不复权日线(全市场某日)
            df = ak.stock_zh_a_hist(
                symbol="", period="daily",
                start_date=date_str, end_date=date_str,
                adjust=""    # 不复权
            )
            if df is None or len(df) == 0:
                continue

            # 转 Polars + 写分区 parquet
            partition_dir = DATA_DIR / f"date={date_str[:4]}-{date_str[4:6]}-{date_str[6:]}"
            partition_dir.mkdir(parents=True, exist_ok=True)

            pl.from_pandas(df).write_parquet(
                partition_dir / "data.parquet",
                compression="zstd",
                use_pyarrow_extension_array=True,
            )
            print(f"  ✅ {date_str}: {len(df)} 行")
            time.sleep(1)    # 避免东财封 IP
        except Exception as e:
            print(f"  ❌ {date_str}: {e}")

fetch_and_store()

查询:Polars 谓词下推的正确姿势

import polars as pl

# ❌ 反模式:全量读再过滤(L10 已讲)
# df = pl.read_parquet("data/daily/**/*.parquet").filter(...)

# ✅ 正确:scan_parquet + 谓词下推
lf = (
    pl.scan_parquet("data/daily/**/*.parquet")
      .filter(
          (pl.col("日期") >= pl.date(2024, 1, 1)) &    # 时间过滤 → 分区裁剪
          (pl.col("收盘") > 0)                          # 数据质量
      )
      .group_by("股票代码")
      .agg(
          pl.col("收盘").mean().alias("avg_close"),
          pl.col("成交量").sum().alias("total_vol"),
      )
      .sort("total_vol", descending=True)
)

df = lf.collect()    # 触发执行
print(df.head())

查询:DuckDB SQL 直查 parquet

import duckdb

# DuckDB 原生支持 hive 分区,谓词自动下推
result = duckdb.sql("""
  SELECT 股票代码, AVG(收盘) AS avg_close, SUM(成交量) AS total_vol
  FROM read_parquet('data/daily/**/*.parquet', hive_partitioning=1)
  WHERE 日期 >= DATE '2024-01-01'      -- 自动裁剪分区
    AND 收盘 > 0
  GROUP BY 股票代码
  ORDER BY total_vol DESC
""").df()
print(result.head())
两者对比

Polars(代码思维,rolling/cross-sectional 因子计算更顺手)+ DuckDB(SQL 思维,ASOF JOIN、ad-hoc 探索更顺手)。两者都免费(MIT)、都有 Python 3.13 wheel、都能查 parquet。不需要二选一,可以并存。

6. 引擎选型决策(10 分钟)

⚠ ArcticDB 散户要不要上

ArcticDB(Man Group 开源)专为量化设计、版本化数据、Python 原生,听起来很美。但:
① 许可是 BSL 1.1(非纯 Apache,2 年后才转 Apache,有 Additional Use Grant);
② 自带存储格式(LMDB/S3),不是 parquet 替代,引入运维复杂度;
③ 散户规模(DuckDB 完全能 hold)用 ArcticDB 是过度工程。
什么时候考虑:分钟级 10 年+、需要符号中心 key-value 访问、要版本化因子。

散户量化,5000 股 × 5 年分钟级数据(~1.5B 行,~50GB),单机 16GB 内存。推荐栈?
  • 纯 Pandas(够熟,内存够)
  • DuckDB + 按 date 分区 parquet
  • KDB+(行业标准)
  • SQLite(轻量)
答案:B。1.5B 行 ~50GB,DuckDB 设计就是为大于内存的 OLAP:按 date 分区后,每次扫描一个分区(~1.2M 行)远小于内存,谓词下推 + 列裁剪让扫描高效。16GB 内存足够(分区是关键)。A(Pandas)单线程 + eager + 全量加载,50GB 直接 OOM;C(KDB+)tick 场景才值,分钟级杀鸡用牛刀 + 商业授权贵;D(SQLite)行存 + 无并行,分析慢。
AKShare 拉某股 5 年历史 qfq 价格做回测,最稳妥的做法是?
  • 每次回测都现拉,保证最新
  • 拉一次存下来,永久不变
  • 存不复权+复权因子,回测时现算 hfq
  • 存 qfq,每周全量刷新
答案:C。存"不复权 + 复权因子"是最稳健方案:不复权价格永不变,复权因子分红时更新,回测时按需算 hfq(后复权,锚定 IPO 日,历史稳定)。A 慢且每次结果可能变(qfq 时变性);B 错(qfq 会随分红变);D 浪费且仍有时变性。这是可复现回测的工程基础。

本课要点回顾

  1. 量化和普通 OLAP 不同:主轴是 symbol×timestamp,写少读多,因子列爆炸。
  2. A股数据首选 AKShare(MIT,免费);Tushare Pro 非商业许可,商业慎用。
  3. 四个数据陷阱:qfq 时变性 / 存活偏差 / 节假日 / 除权除息。必须防护,否则回测失真。
  4. parquet 按 date 单层分区,row group 按 symbol 排序;不要按 symbol 分区(5000 小文件灾难)。
  5. Polars(代码)+ DuckDB(SQL)并存:两者都 MIT、都有 3.13 wheel、都查 parquet,不需要二选一。
  6. 规模 → 引擎:日级 → 文件够;分钟级 5 年 → DuckDB + 分区;分钟级 10 年+ → ArcticDB;tick → ClickHouse。
  7. 散户不要上 ClickHouse/ArcticDB:无 tick + 单用户场景,嵌入式方案是降维优势(零运维)。
